# Repo Learning Orchestrator

一个帮助用户“逐层看懂代码仓库”的学习型产品设计说明。

它的目标不是帮用户读完一个 repo，而是帮用户每次多懂一层 repo。

## 核心理念

传统的代码分析工具更像“总结器”：

- 给出目录摘要
- 给出文件说明
- 给出长篇分析

但真正能让用户坚持学下去的，不是更多信息，而是持续获得理解感。

所以，这个产品的核心不是“提供信息”，而是“推动理解”。

一句话定义：

> 不帮用户读完 repo，而帮用户每次多懂一层 repo。

## 用户为什么学不下去

对代码框架感兴趣的人，通常不是因为没兴趣而放弃，而是因为很快陷入以下问题：

### 1. 信息量过大

打开一个仓库后，用户面对的是：

- 很多目录
- 很多文件
- 陌生依赖
- 隐含约定

他不知道第一步应该看什么。

### 2. 缺少即时反馈

读了很久代码，却无法回答：

- 这个系统到底怎么跑起来
- 哪些模块是核心
- 自己到底有没有真的看懂

没有反馈，就很难坚持。

### 3. 学习路径不自然

很多人一上来就从单个文件开始读，但理解项目更自然的顺序通常是：

1. 先理解项目解决什么问题
2. 再理解系统的主要模块和数据流
3. 再理解核心主流程
4. 最后进入局部实现细节

如果顺序错了，就会一直迷路。

### 4. 没有任务感

纯“阅读代码”很容易空转。人更容易坚持的是：

- 带着问题读
- 带着任务读
- 带着验证读

## 产品应该解决什么问题

这个产品不应该只是：

> 帮用户总结项目

而应该是：

> 帮用户一步步拆开一个系统，让他不断产生“我又懂了一层”的感觉

因此，学习体验要从：

- 被动看总结

转为：

- 主动完成理解任务

用户能否坚持，取决于三件事：

1. 他知道自己现在在学什么
2. 他知道下一步该看什么
3. 他能感受到自己刚刚真的学会了一点东西

## 学习模型：分层递进

最适合代码框架学习者的方式，不是长篇总结，而是分层学习。

### 第一层：地图层

目标：先建立方向感。

输出内容：

- 这个项目是干什么的
- 核心模块有哪些
- 请求、数据、状态如何流动
- 每个目录大致负责什么

这一层不追求细节完整，而是帮助用户先“看见全局”。

### 第二层：主干层

目标：抓住系统骨架。

输出内容：

- 程序入口在哪里
- 路由或调度从哪里开始
- controller / service / repository 如何串起来
- 配置在哪里生效
- 数据最终落到哪里

这一层只讲主流程，不讲全部文件。

### 第三层：局部深入层

目标：按专题理解关键设计。

可选专题示例：

- 登录鉴权怎么做
- 配置加载怎么做
- 数据库访问怎么抽象
- 任务调度怎么设计
- 前后端通信怎么组织

这一层是按主题学习，而不是按文件学习。

### 第四层：复述与迁移层

目标：把“看懂”转化为“会讲、会用”。

输出内容：

- 你能不能复述这个项目的架构
- 你能不能解释为什么这样设计
- 你能不能把这种设计迁移到别的项目

真正的能力感，来自用户可以自己讲出来。

## 最适合坚持的机制：小闭环

这个产品不应该做成“完整 repo 精讲课”，而应该由很多小闭环组成。

每次学习只完成一个明确目标，例如：

- 今天只搞懂入口和启动流程
- 今天只搞懂请求如何走到数据库
- 今天只搞懂状态管理和事件流
- 今天只搞懂配置系统

每个小节都应该在 5 到 15 分钟内产生一个明确结果：

> 我现在能讲清楚这一块了

## 最容易让人坚持的内容组织方式

### 1. 问题驱动

比起直接讲“这是一层 service”，更应该问：

- 为什么这里要分 service 和 repository
- 为什么这个项目按业务域组织，而不是按技术层组织
- 为什么这里使用依赖注入
- 为什么这里用 event bus，而不是直接调用

“为什么”比“是什么”更容易激发持续学习。

### 2. 路径驱动

不要让用户自由探索全部代码，而是明确给出路径：

- 第一站看哪里
- 第二站看哪里
- 第三站看哪里

例如：

1. 先看 `main.ts`
2. 再看 `router`
3. 再看 `service`
4. 最后看数据库层

路径越明确，越容易坚持。

### 3. 对比驱动

通过对比理解设计选择：

- 这个 repo 为什么不是按技术层拆分
- 它和常见 Express 项目有什么不同
- 为什么这里使用 event bus

对比能让架构决策变得更有意义。

### 4. 任务驱动

学习过程中应配套极轻量的“侦探任务”：

- 找出项目真正的入口
- 画出一次请求的流向
- 找出配置注入点
- 找出数据库连接初始化时机

这不是考试，而是帮助用户主动参与理解。

## 用户坚持的心理机制

### 1. 可见进度

系统应该持续告诉用户当前进展，例如：

- 已理解：项目目标
- 已理解：目录结构
- 已理解：主流程
- 下一步：状态管理

### 2. 即时奖励

每完成一块，都要给出与能力绑定的反馈，例如：

- 你已经能解释启动流程了
- 你已经掌握核心请求链路
- 你已经识别出这个 repo 的分层模式

### 3. 难度刚好

理想状态不是“很简单”，而是：

> 有点难，但我能跟上

所以系统需要动态控制解释深度，避免一次输出过长、过重的分析。

## 后端应该扮演的角色

后端不只是代码解析器，更是学习编排器。

它至少要完成三类事情：

### 1. 建立项目知识图谱

识别：

- 入口点
- 核心模块
- 模块依赖
- 主调用链
- 关键设计模式
- 值得深入学习的局部

### 2. 判断用户当前阶段

用户可能处于不同状态：

- 完全新手
- 只想快速理解架构
- 想深入某个模块
- 想学习某种模式

### 3. 给出“下一步最值得学什么”

这一步最关键。真正帮助用户坚持的，不是把所有内容一次性给完，而是明确告诉他：

> 下一步最该看什么，以及为什么现在看它最有价值

## 一个适合产品实现的学习状态机

整个学习过程可以抽象为四个状态：

### 状态 1：定向

这个项目是什么，应该从哪里开始看。

### 状态 2：建模

这个系统有哪些模块，它们如何连接。

### 状态 3：追踪

跟着一个真实流程走一遍。

### 状态 4：内化

用户能够复述、比较并迁移这种架构。

产品每次只需要推动用户从当前状态往下一个状态前进一步。

## 推荐的产品交互形式

最适合代码框架学习者的形式不是传统教程，而是：

> 地图 + 路线 + 任务 + 解释

### 地图

提供全局感。

### 路线

告诉用户先学哪、后学哪。

### 任务

给出具体观察目标和验证动作。

### 解释

在用户卡住时提供抽象总结和必要说明。

这四者结合，比单独提供“解释”更能让人坚持。

## 产品设计中最应该避免的三件事

### 1. 一上来给超长总结

用户可能觉得内容很强，但并不会真正学进去。

### 2. 按文件逐个讲

这更像文档，不像学习系统。

### 3. 把重点放在完整覆盖

学习产品不需要完整覆盖，而需要持续制造“我又懂了一层”的体验。

## 后端的五个核心问题

后端不应该只是生成一篇报告，而应该持续回答这五个问题：

1. 这个项目最重要的骨架是什么
2. 对当前用户来说，下一步最值得理解的部分是什么
3. 怎样把这一部分解释得刚好不难
4. 怎样验证用户真的理解了
5. 怎样让他自然进入下一步

## 判断一个功能是否值得做的标准

设计任何功能时，都问一句：

> 这个功能是在提供信息，还是在推动理解？

只有后者，才真正有机会让用户坚持下去。

## 适合作为 MVP 的能力清单

如果要把这个方向做成一个最小可用产品，建议优先实现：

1. 自动识别项目入口、核心目录和主流程
2. 生成“地图层”说明
3. 为用户推荐下一步阅读路径
4. 给出一个轻量理解任务
5. 在任务后给出能力化反馈

## 产品一句话总结

这是一个面向代码框架学习者的 repo 学习系统。

它不追求覆盖全部代码，而追求让用户在一次次小闭环中，持续建立对项目结构、主流程和设计思路的理解。
